# 評価基準全体を満たすミニマルアプリケーション条件

作成日: 2026-06-20

## この文書の位置づけ

この文書はTODO 4「評価基準全体を満たせるミニマルアプリケーションの条件を定義する」の成果物である。

ここでは、まだ作成するアプリケーションの題材そのものは確定しない。TODO 5で題材を決める前に、どのような性質・機能・開発成果物を持つアプリでなければ評価基準を網羅できないかを定義する。

## 前提

- 評価基準は有効項目として53件ある。
- `その他開発知見 / トランザクション処理・個人情報マスキング` はExcel上で評価基準本文が空欄のため要確認だが、後続設計では可能な限り吸収できる余地を残す。
- 評価対象は「画面上の機能」だけではない。Railsアプリ本体、テスト、Docker、CI、リンター、Git運用、API仕様、評価資料、コード上の設計証跡まで含めて評価基準を満たす。
- ただし、評価官から質問されたときに「資料にだけ書いてある」状態にはしない。できる限り、動くアプリ・コード・テスト・設定ファイル・ログ・API仕様のいずれかで実物を示せる状態にする。

## 結論

評価基準全体を満たすには、単純なCRUDアプリでは不足する。

必要なのは、次の性質を持つミニマルな業務ワークフロー型Railsアプリである。

- 複数ロール・複数権限を持つ
- 認証方式を画面向けとAPI向けで分けられる
- ドメイン上の状態遷移・承認・差し戻し・通知がある
- 一覧検索、集計、詳細、履歴、監査ログがある
- CSV/Excel取込、PDF/Excel出力がある
- 非同期ジョブ、外部API連携、リトライ、冪等性がある
- REST API、GraphQL、WebSocketまたはリアルタイム通知を持つ
- DB性能、ロック、N+1、インデックス、EXPLAINを説明できる
- テスト・CI・Docker・リンター・依存管理・Git運用まで含む
- Docker構築だけでなく、PCストレージ負荷を抑えるためのローカル構築手順も持つ

この条件を満たすアプリなら、評価基準53件を「アプリ機能」「コード設計」「テスト」「開発環境」「資料」の組み合わせで網羅できる。

## 網羅に必要なアプリケーション条件

### 1. 業務ドメイン条件

アプリ題材は、以下の業務要素を自然に含められるものにする。

- ユーザー、ロール、権限、テナントまたは所属組織がある
- 作成、申請、承認、差し戻し、完了、取消などの状態遷移がある
- 一覧、詳細、登録、編集、削除、検索、集計がある
- コメント、添付、通知、履歴、監査ログがある
- 大量データの取込・出力がある
- 外部サービス連携がある
- 非同期で処理した方が自然な処理がある

この条件を満たさない題材は、Rails Lv2とバックエンド共通スキルを自然に載せづらい。

### 2. 認証・認可条件

アプリは次を持つ必要がある。

- 画面向けのログイン、ログアウト、サインアップ
- パスワードハッシュ化、バリデーション、認証エラー処理
- API向けJWTアクセストークン
- リフレッシュトークンとローテーション
- Cookie/session認証とJWT認証の使い分け説明
- 外部IDPまたはローカルで再現可能なOAuth相当の認証デモ
- ロールベース認可
- テナントスコープを含む認可
- Pundit等によるポリシー定義
- 認可のユニットテスト

これにより、Rails認証・認可、バックエンド共通の認証・認可、OWASP Broken Access Control、Authentication Failuresをまとめて説明できる。

### 3. ドメインロジック条件

アプリは、モデルに処理を寄せすぎず、サービス層・ユースケース層・値オブジェクトを持つ必要がある。

最低限、次を含める。

- Factoryパターン
- 依存性注入
- Duck Typing
- NotImplementedErrorを使った抽象基底クラス
- Concernまたはモジュール分離
- StructまたはDataクラス相当の不変値オブジェクト
- freezeされた定数
- ガード節、早期リターン
- SRPを満たす小さいメソッド
- カスタム例外と例外ハンドリング方針
- ブロックを受け取る共通処理
- lambda/procの使い分け説明ができる箇所
- Enumerableによる複雑データ加工
- メタプログラミングを限定的かつ追跡可能に使う箇所

この条件により、Ruby基礎文法、オブジェクト指向、標準クラス・機能の評価基準をアプリ内部の実装で示せる。

### 4. ActiveRecord・DB条件

アプリは、単純な1対多だけではなく、少し複雑なデータ構造を持つ必要がある。

最低限、次を含める。

- 複数モデルの関連
- STIまたはポリモーフィック関連
- カスタムバリデータ
- コールバックと、例外時のロールバック制御
- トランザクション処理
- 楽観的ロックまたは悲観的ロック
- インデックス設計
- EXPLAINで確認できる検索クエリ
- N+1を発生させうる一覧と、includes/preload/eager_loadによる改善
- find_eachまたはeach_sliceによる大量データ処理
- RidgepoleのSchemafile/dry-run/applyを使った安全なDBスキーマ変更方針

これにより、ActiveRecord、DB、トランザクション、性能、ロックの評価基準をカバーする。

### 5. 検索・一覧・集計条件

アプリは、評価官に見せやすい検索・一覧・集計機能を持つ必要がある。

最低限、次を含める。

- 複数条件検索
- 複数モデルをまたぐ検索
- Ransack等の検索自由度制御
- ページネーション
- カーソル方式またはlimit/offset方式の説明
- SQL発行回数を説明できる一覧画面
- インデックス有無によるクエリ差分を示せる設計

検索機能は、ActiveRecord、DB性能、APIページネーション、N+1、セキュリティの説明に使いやすいため、発表上も重要である。

### 6. ファイル・帳票条件

アプリは、ファイル入出力と帳票を持つ必要がある。

最低限、次を含める。

- CSVまたはExcelの取込
- 大容量ファイルを一括読み込みしない逐次処理
- Excel出力
- PDF出力
- 取込時のバリデーションエラー表示
- 取込失敗時の例外種別とハンドリング
- バックグラウンドジョブ化できる構造

これにより、Rubyファイル操作、PDF/Excel連携、大量処理、例外処理、非同期処理をまとめて説明できる。

### 7. 非同期・外部連携・通知条件

アプリは、外部依存を持つ処理を直接コントローラに書かない。

最低限、次を含める。

- ActiveJob
- Redis等を使うジョブ実行環境
- 冪等性キーまたは処理済み管理
- retry_on/discard_on等の再試行・失敗時処理
- HTTPクライアントクラス
- open_timeout/read_timeout
- 5xx/timeout時のexponential backoff
- Slack/Google Calendar/Mockなど差し替え可能な外部連携クラス
- DIによりテスト時に外部依存を差し替えられる構造

これにより、HTTPクライアント、外部API連携、非同期、テストのモック/スタブ、疎結合を説明できる。

### 8. API・通信条件

バックエンド共通スキルを満たすには、画面だけでは足りない。

最低限、次を含める。

- `/api/v1` 配下のREST API
- 統一エラーレスポンス
- APIバージョニング
- APIページネーション
- OpenAPI/Swagger定義
- GraphQL Query
- GraphQL Mutation
- WebSocketまたはActionCableによるリアルタイム通知
- gRPCについては、実装必須ではなく概要説明と `.proto` 例を資料またはdocsに残す

REST、GraphQL、WebSocket、OpenAPIは、評価官から「標準的なバックエンド知識」として聞かれやすい。Rails画面中心のアプリでも、このAPI層は意図的に作る必要がある。

### 9. セキュリティ条件

アプリは、OWASP観点をコードで説明できる必要がある。

最低限、次を含める。

- 認証・認可チェック
- テナントスコープによるデータ分離
- SQLインジェクションを避けるクエリ実装
- パスワードハッシュ化
- 秘密情報の環境変数化
- ログへの機密パラメータ出力抑制
- CSRF対策
- session fixation対策
- 入力値長の制限
- ReDoSを避ける正規表現設計
- 監査ログ

セキュリティは試験官が深掘りしやすい領域なので、単に「対策しています」ではなく、攻撃パターンと対策箇所をセットで説明できるようにする。

### 10. 開発基盤・運用品質条件

アプリ本体だけでなく、開発成果物として次を持つ必要がある。

- Dockerfile
- docker-compose.yml
- Rails、DB、Redisをローカルで一括起動できる構成
- 環境変数の外部化
- RuboCop等のリンター設定
- formatter設定
- pre-commit hookまたはovercommit設定
- bundle audit等の脆弱性チェック手順
- GitHub Actions等のCI設定
- テスト用DB/スキーマ分離
- FactoryまたはFixture
- request/model/policy/system/jobテスト
- 外部APIモック/スタブ
- Git運用メモ
- Dockerを使わないローカル構築手順
- `bin/setup`、`bin/dev` 等の軽量な起動手順

Gitのstash/rebase/reset/revert/cherry-pickなどはアプリ機能として実装するものではないため、リポジトリ運用資料・コミット履歴・説明資料で証明する。

## 網羅判定の基準

各評価基準には、最終的に以下のいずれか複数の証跡を紐づける。

- 動く画面
- APIエンドポイント
- Railsモデル・サービス・ジョブ・ポリシー等のコード
- テストコード
- Docker/CI/lint等の設定ファイル
- OpenAPI/GraphQL schema等の仕様
- ログ・監査履歴
- 発表資料
- 評価基準対応表

重要なのは、全項目を「資料で説明できる」だけでなく、可能な限り実物の成果物に接続することである。

## 試験官の言葉から注力すべき箇所

Wordメモから見ると、発表で特に注力すべき箇所は次の通り。

### 最優先で見せるべき領域

1. Whyを説明できる設計判断
   - 「どう実装したか」だけではなく、「なぜその設計にしたか」を説明する。
   - 例: Fat Modelを避けてサービス層に分けた理由、DIにした理由、Punditで認可を分けた理由。

2. AI出力を自責でレビューできること
   - AIが出したコードや設計をそのまま採用せず、アーキテクチャ、凝集度、結合度、セキュリティ、テスト容易性でレビューした証跡を見せる。
   - 実装後に `AIレビュー記録` を資料またはdocsに残すと説明しやすい。

3. アーキテクチャ・高凝集・疎結合
   - バックエンド共通では概念理解が重視されるため、サービス層、値オブジェクト、Repository相当、通知インターフェース、DIを中心に説明する。
   - 「どの層が何に依存しているか」を図で説明できる状態にする。

4. 認証・認可・OWASP
   - Broken Access Control、Authentication Failures、Injection、Cryptographic Failuresは質問されやすい。
   - ロール・テナント・ポリシー・テストを実物で示せるようにする。

5. 具体的な実装証拠
   - 「勉強しました」より「このアプリではこう実装しました」が強い。
   - 発表では画面、コード、テスト、ログ、API仕様を短時間で切り替えて見せる構成にする。

### 次点で厚めに準備すべき領域

6. Rails固有機能とバックエンド標準の差分
   - Railsではどう実装し、一般的なバックエンドではどう説明されるかを比較できるようにする。
   - 例: RailsのActiveJobと一般的なキュー、Punditと一般的なRBAC、ActiveRecordとSQL/EXPLAIN。

7. テスト・CI・品質担保
   - 評価基準上も、実務説得力の面でも重要。
   - 特に認可テスト、APIテスト、ジョブテスト、外部APIスタブ、DBテストを見せられるようにする。

8. DB性能・N+1・ロック
   - 実務的な深掘りを受けやすい。
   - includes/preload/eager_load、EXPLAIN、index、lock_version、FOR UPDATEの説明材料を用意する。

9. API・通信規格
   - REST、GraphQL、WebSocket、OpenAPIを「動くもの」として持つと、Rails画面アプリに閉じないバックエンド共通知識を示せる。

10. 資料の全網羅と発表の重点化
   - Wordメモでは「資料には全てを載せ、プレゼンでは主張したいところを重点説明する」方針が示されている。
   - そのため、発表資料は重点説明、評価基準対応表は全網羅という二層構成にする。
