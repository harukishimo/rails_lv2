## 分類方針

### Ruby/Rails側

`rails lv2` シートの内容を扱う。シート内にはRuby言語そのものの項目とRails固有項目が混在しているため、要件定義上は以下の2段に分ける。

- Ruby基礎・Ruby標準機能
- Rails固有機能・Rails実務機能

### バックエンド共通側

`Back End 共通` シートの内容を扱う。

シート内の `【共通】` はバックエンド全般の概念・説明能力、`【BE固有】` はバックエンド実装スキルを指す。どちらもRails固有ではなく、バックエンド共通カテゴリとして扱う。

## 1. Ruby/Rails側評価基準

### Ruby / 基礎文法

1. 変数・定数: 定数への破壊的変更のリスクを特定し、freeze 等による不変性の確保をすべき場面を選択できる。
2. 式: 複雑な式を補助変数やメソッドへ分離し、副作用を抑えつつ保守性の高い構造へ変換できる。
3. 条件分岐: ガード節や早期リターンを用い、ネストによる複雑化のリスクを排除した構造へリファクタリングできる。
4. メソッド: 再利用性と責務分離を意識し、単一責任原則（SRP）を保った小さなメソッドへ分解・整理ができる。
5. ブロック: proc や lambda の挙動の違いを説明でき、ブロックを受け取るメソッドを定義できる。
6. ハッシュ/配列: 列挙用メソッドを駆使し、複雑なデータ構造の抽出・加工を効率的に実装できる。
7. 例外処理: 複数種類のエラーを整理し、例外クラスの粒度設計と適切なハンドリング方針を設計・実装できる。

### Ruby / オブジェクト指向

8. クラス: Factoryパターンや依存性の注入（DI）を用い、生成ロジックとインスタンス側の責務を適切に分離し、テスト容易性を高める設計ができる。
9. 継承 / super: 共通処理を基底クラスやConcernに集約し、親の肥大化を避けつつ可変部分を子で実装する設計ができる。また、NotImplementedError等を用いてサブクラスの実装責務を明示する抽象的な設計ができる。
10. ポリモーフィズム: Duck Typingを活用して分岐を排除し、拡張可能かつ呼び出し側の責務を最小化した設計ができる。
11. カプセル化: 外部のコードを変更せずに内部実装を入れ替えられるよう、データへの直接アクセスを避けた設計ができる。
12. モジュール: include/extend/prepend の動作差異から、メソッド探索順序への影響とリスクを説明・制御できる。
13. Struct: Struct の利用による読み取り専用（不変性）のメリットを説明でき、データの受け渡しをカプセル化できる。

### Ruby / 標準クラス・機能

14. 日付・時間: タイムゾーン不整合によるデータ汚染のリスクを特定し、ActiveSupport 等で安全な日時操作を実装できる。
15. ファイル操作: 大容量ファイル読み込み時のメモリ枯渇リスクを説明でき、foreach 等の逐次処理による対策を選択できる。
16. 外部連携（PDF/Excel）: gemを用い、基本的なPDF出力やExcelファイルの読み書きを実装できる。
17. 正規表現: 複雑な正規表現（肯定的先読み等）を駆使し、「ReDoS（正規表現の脆弱性）」のリスクと対策を説明できる。
18. 範囲（Range）: cover? と include? の動作差異（連続値か離散値か）を説明でき、適切なメソッドを選択できる。
19. 並行処理（スレッド）: Thread を用いた基本的な並行実行ができる。スレッド間競合（レースコンディション）のリスクを特定し、Mutex 等で共有資源を保護できる。
20. gem / bundler: 破壊的変更のリスクを特定し、gemのアップデートに伴う構文変更（DEPRECATION WARNINGへの対応等）や、既存コードの修正を伴うバージョン管理ができる。
21. メタプログラミング: メタプログラミングを用いた共通化を行いつつ、追跡可能性（保守性）を損なわない設計ができる。

### Rails / ActiveRecord

22. CRUD: N+1問題などのクエリ効率リスクを特定し、includesやjoinを用いて改善できる。
23. 一括処理: 大量データ処理時の負荷（メモリ・CPU）を特定し、find_each や非同期処理を適切に選択・実装できる。
24. マイグレーション: Strong Migrations 等で危険なDDLを検知し、データパッチ作成やリリース順序を安全に設計できる。
25. バリデーション: カスタムバリデータを作成し、ドメイン固有の複雑な制約を実装できる。
26. コールバック: コールバック内例外によるロールバックの影響を制御し、明示的なエラーハンドリングを実装できる。
27. 関連付け: ポリモーフィック関連付けやSTIを適切に設計・実装し、複雑なドメインモデルを表現できる。
28. 複雑なクエリ: preload/eager_load/includes の挙動の差分から、SQL発行回数とメモリ負荷のトレードオフを最適化できる。

### Rails / View・Controller

29. View / Decorator: Railsの標準機能（UJS/Turbo等）を超えた要件に対し、JSを用いたカスタムの非同期通信を設計・実装し、保守性・拡張性の高いフロントエンド・バックエンド連携を実現できる。
30. Controller: ロジックをサービス層等へ委譲し、機密パラメータのログ出力抑制等のセキュリティ対策を実装できる。

### Rails / 認証・認可

31. 認証: Web/SPA/モバイル等、クライアントに応じた認証方式（Cookie/JWT等）を使い分け、ソーシャル連携や外部IDP連携など、実務で頻出する外部認証機能を実装できる。
32. 認可: 認可処理（Punditなど）を適切に設計し、複雑な条件でのアクセス制御（テナントスコープ、ロールベースなど）を実装できる。
33. 検索: インデックスの利用やクエリ最適化に加え、ransackが提供する自由度を制御し、設計された構造を保ちつつ複数条件・複数モデルを跨ぐ複雑な検索ロジックを効率的に実装できる。

### Rails / 実務機能

34. 非同期・タスク: ActiveJobを用いた非同期処理の定義と実行、およびJobの冪等性や再試行を考慮した基本的なエラーハンドリングを実装できる。
35. HTTPクライアント / 外部API連携: 特定の外部APIに対するクライアントクラス（ベースURL・認証情報・接続設定を保持）を設計できる。タイムアウト設定（open_timeout/read_timeout）や一時的なエラー（5xx/タイムアウト）に対するリトライ処理（最大3回、exponential backoff等）を実装できる。
36. ログ / その他: ログローテートや保管方法を要件に合わせて設定し、有事の際の追跡可能性を担保できる。

### Rails / テスト

37. テスト: CI環境への統合、および仕様変更に強いテスト構成（ユニット/結合/システム）を設計・実装できる。また、テストコードの共通化、テスト用ユーティリティやseedデータの用意により、チーム開発の効率とテスト実行環境の整備を行える。

### Rails / 基盤

38. 基盤構築: 認証、権限、複数方向のログインを含むアプリを0から作成できる。チームメンバーの開発を手助けするような機能（scaffoldのカスタマイズ、マイグレーションの整備、税計算などのutil）を揃えた基盤を作成できる。

## 2. バックエンド共通側評価基準

1. AI / AIリテラシー: 【共通】Lv.1への指導ができる + AIの出力を自責でレビューできる。
2. システム要件定義/基本設計: 【共通】システムの技術要件や非機能要件を詳細に定義し、技術的実現性（工数、難易度、リスク）を評価できる。要件に優先順位（Must/Should/Could/Won't等）を付け、プロジェクトスコープの調整を提案できる。利害関係者との協議を通じて、要件の合意形成を図れる。
3. 環境構築 / Docker: 【共通】Dockerを使って効率的な開発環境を構築し、チーム全体で共有できる。Dockerfile、docker-compose.yml、ボリュームマウント、環境変数外部化を扱える。
4. アーキテクチャ / アーキテクチャ構成: 【共通】クリーンアーキテクチャ、DDD、オニオンアーキテクチャ、ヘキサゴナルアーキテクチャ等の設計思想を説明し、既存コードベースでその思想に沿った機能改修・追加ができる。
5. アーキテクチャ / 凝集と結合: 【共通】高凝集・疎結合を説明できる。【BE固有】インターフェースによる抽象化とDIを説明し、密結合の問題を指摘できる。
6. セキュリティ / 認証: 【共通】アクセストークン・リフレッシュトークン、認証フローを説明できる。【BE固有】メール/パスワード認証、JWT発行・検証、リフレッシュトークンローテーションを実装できる。
7. セキュリティ / 認可: 【BE固有】権限テーブルを設計し、標準的な認可ライブラリを導入し、ロールごとのCRUD権限をポリシーとユニットテストで表現できる。
8. セキュリティ / OWASP: 【共通】OWASP Top 10の存在を知り、Broken Access Control、Cryptographic Failures、Injection、Authentication FailuresなどBE関連脆弱性を説明できる。
9. 通信規格: 【共通】REST vs GraphQL、WebSocket vs HTTPを説明できる。【BE固有】RESTful URL設計、APIバージョニング、ページネーション、統一エラーレスポンス、OpenAPI/Swagger、gRPC、GraphQL、WebSocket実装を扱える。
10. フォーマッター/リンター: 【共通】フォーマッター/リンターを導入し、設定ファイル、pre-commitフック、プロジェクト要件に応じたルールカスタマイズができる。
11. ビルドツール: 【共通】エントリーポイント、出力先、環境変数注入、本番ビルド最適化、ホットリロード/ライブリロードを説明できる。
12. パッケージマネージャー: 【共通】バージョン指定、脆弱性チェック、依存ライブラリの定期アップデート方針を説明できる。
13. Git: 【共通】コンフリクト、stash、rebase、reset/revert、cherry-pick、誤ったcommit/pushからの復旧手順を説明し使い分けられる。
14. テスト / ユニットテスト: 【共通】APIエンドポイントテストを実装できる。【BE固有】モック/スタブ、DBテスト、トランザクションロールバック、テストDB/スキーマ分離、Factory/Fixtureを扱える。
15. データベース: 【BE固有】EXPLAINでボトルネックを特定し、インデックス追加やクエリ書き換えでチューニングできる。機能追加に伴うDB設計、楽観的/悲観的ロック、FOR UPDATE、バージョンカラムを説明・実装できる。

## 3. Wordメモから抽出した評価官の注視ポイント

### 評価の方向性

- 評価基準は、評価者によるブレを減らすために明確化された。
- 丸暗記やHowだけではなく、「なぜそれを使うのか」というWhyを深掘りできるように変更されている。
- 単なる知識確認ではなく、理由・判断基準・設計意図を説明できることが重視される。

### AI・バックエンド共通スキル

- AI関連スキルは一部の人だけではなく、全体に関わるバックエンド共通スキル側へ移動・追加された。
- AIの出力をそのまま信じるのではなく、自責でレビューし、良し悪しを判断できることが重視される。
- AI出力のレビュー能力を問うために、アーキテクチャ、凝集度、結合度などの概念理解が重要視されている。

### バックエンド共通の評価観点

- クイズ形式の知識確認ではなく、概念理解・思想理解が重視される。
- 特にアーキテクチャ、高凝集・疎結合、セキュリティ、正規表現などは、他言語でも通用する標準的な内容として見られる。
- 言語間の平準化が意識されており、Railsだけの知識ではなく、一般的なバックエンド標準との対応を説明できる必要がある。

### プレゼンテーション方針

- 資料には全評価基準を載せる。
- 15分の発表では、全項目を逐一説明するのではなく、自分が重要だと考える項目を選んで重点的に説明する。
- 重点説明から外した項目も、質問された場合に資料・アプリ・コードで示せる状態が必要。
- 「全部に対して説明していたら時間がなくなる」という課題が明示されているため、網羅資料と発表用ピックアップを分ける方針は妥当。

### 実績・具体例の重視

- 手元で勉強した知識だけより、実際に使った・作った・判断した具体例の方が説得力がある。
- 抽象論だけではなく、「具体的に何をしてきたか」「どの機能でどう実装したか」を整理しておく必要がある。
- 実務で使っている技術が標準と違う場合でも、標準的な知識と自分のプロジェクトとの差分を説明できれば評価上問題ないという発言がある。

### アプリ開発方針への示唆

- 複数項目を盛り込んだアプリを1つ作って説明する方針は、メモ内の発言と整合している。
- 作成するアプリは、評価基準全体を実装・設計・テスト・資料で裏付けられる必要がある。
- 発表では、評価基準の全網羅ではなく、Whyを説明しやすく評価官が深掘りしやすい項目を重点化するのがよい。
