# SkillEvidenceHub 要件定義書

作成日: 2026-06-20

## 1. 目的

`SkillEvidenceHub` は、受験対象マスタを管理し、受験者が自分で取得・学習したい言語/Lvに対して受験表明を行い、その受験単位に紐づく提出物、Markdownコメント、レビュー、面談、合格判定、取得資格反映を管理する社内評価支援アプリケーションである。

評価基準本文や詳細カテゴリは、別の社内ナレッジシステムで管理される前提とする。このアプリケーションでは、言語、Lv、外部ナレッジ参照、レビュー状態、提出物、コメント、面談調整を管理する。

また、受験表明に紐づく評価面談応募を扱い、評価官マスタから面談対応評価官を割り振りフォームの初期値として自動フィルインし、対象評価官が割り振りを確定した後、日時承認後にGoogle Calendar予定を作成する。面談後に評価官が合格判定を行った場合、ユーザーの取得資格へ反映し、受験単位をクローズする。

このアプリケーション自体は、Ruby/Rails Lv2およびバックエンド共通の評価基準を満たす実装例としても扱う。

## 2. 背景

社内評価では、複数の技術領域・言語・Lvにまたがる評価基準が存在する。評価基準の本文や詳細カテゴリは社内ナレッジシステム側で管理される。受験者は評価官から一方的に評価基準を指定されるのではなく、自分が勉強・取得したい言語/Lvを選択し、学習・実装・提出を行う。

受験表明は、レビュー依頼や面談応募より大きい親単位として扱う。受験表明を登録すると、その受験単位に紐づけて評価基準レビュー依頼を出せるようになり、評価面談にも応募できるようになる。

評価官は、自分が対応可能な評価スキルに関する受験表明・レビュー依頼のみを確認し、必要に応じてコメント、差し戻し、承認を行う。レビューは対応可能な評価官のうち誰か1人が対応した時点で、レビュー依頼全体のステータスを更新する。

評価面談は受験表明に紐づく子フローとして扱う。受験者が面談応募を登録し、システムが評価官マスタから面談対応数の少ない評価官を割り振りフォームの初期値として自動フィルインする。この候補は保存前の初期値であり、対象評価官が指定の評価官へ変更して確定できる。評価官確定後、希望日時を指定し、承認されたらGoogle Calendar予定を作成する。面談後に評価官が合格判定を登録すると、取得資格へ反映し、受験をクローズする。

## 3. ゴール

- 受験対象マスタとして言語、Lv、外部ナレッジ参照を管理できる
- 受験者が受験対象マスタから受験したい言語/Lvを選択し、受験表明を登録できる
- 受験表明に紐づいてレビュー依頼と面談応募を作成できる
- 受験者がMarkdown形式の任意コメントで学習・実装・アピール内容を登録できる
- 受験者が提出物ファイルまたはGitHubリポジトリURLを登録できる
- 評価官が自分の対応可能評価スキルに関する提出物をレビューし、コメント、差し戻し、承認できる
- 評価面談応募、評価官自動フィルイン、評価官割り振り確定、希望日時承認、Google Calendar予定作成ができる
- 面談後の合格判定をユーザーの取得資格へ反映し、受験をクローズできる
- Slackでレビュー依頼、差し戻し、承認などを通知できる
- 管理者が受験対象、評価期、ユーザー、ロール、権限、評価官マスタ、対応可能評価スキルを管理できる
- ローカル構築とDocker構築の両方に対応する
- アプリ自体が評価基準全体を満たす根拠になる

## 5. 想定ユーザー

### 5.1 受験者

- 受験対象マスタから受験したい言語/Lvを選択する
- 言語/Lvに対する受験表明を登録する
- 提出物ファイルをアップロードする
- GitHubリポジトリURLを登録する
- Markdown形式の補足説明・アピールコメントを登録する
- レビュー依頼を提出する
- 受験表明に紐づけて評価面談へ応募する
- 差し戻しに対応する

### 5.2 評価官

- 提出物を確認する
- 受験対象と提出内容に対してコメントする
- 必要な確認事項をレビューコメントとして残す
- 差し戻し、承認、却下を行う
- 面談時に確認したい観点を残す
- 自分が対応可能な評価スキルの受験表明・レビュー依頼だけを確認する
- 評価面談の希望日時を承認または差し戻す
- 面談対応評価官の自動フィルイン結果を必要に応じて変更し、割り振りを確定する
- 面談後に合格/不合格を判定する

### 5.3 管理者

- 受験対象マスタを管理する
- 技術領域、言語、フレームワーク、Lv、外部ナレッジ参照を管理する
- 評価官マスタを管理する
- 評価官ごとの対応可能評価スキルを管理する
- 評価官ごとの面談対応数を確認する
- 評価期を管理する
- 受験者と評価官を管理する
- ロールと権限を管理する
- Google Calendar連携設定を管理する
- 監査ログを確認する

## 6. 業務フロー

### 6.1 受験全体

1. 管理者が受験対象マスタをExcel/CSVから取り込む
2. 管理者が評価期、受験者、評価官マスタ、評価官の対応可能評価スキルを設定する
3. 受験者が受験対象マスタから言語/Lvを指定し、受験表明を登録する
4. 受験表明により、同じ受験単位でレビュー依頼と面談応募を作成できるようになる
5. 受験者が受験表明に紐づくレビュー依頼を作成し、提出物ファイルまたはGitHubリポジトリURL、Markdownコメントを登録する
6. 対応可能な評価官が評価官ページでレビュー依頼を確認する
7. 評価官のうち誰か1人がコメント、差し戻し、承認、却下を登録し、レビュー依頼全体のステータスを更新する
8. 受験者が受験表明に紐づく評価面談応募を登録する
9. システムが評価官マスタから面談対応評価官を割り振りフォームの初期値として自動フィルインする
10. 対象評価官が必要に応じて割り振りを編集し、指定の面談対応評価官へ変更する
11. 受験者が希望日時を指定する
12. 評価官が希望日時を確認し、問題なければ承認する
13. 承認後、受験者・評価官のメールアドレスを使ってGoogle Calendar予定を作成する
14. 面談後に評価官が合格/不合格を判定する
15. 合格の場合はユーザーの取得資格に反映し、受験単位をクローズする

### 6.2 評価基準レビュー

レビュー依頼は受験表明に紐づく子フローとして扱う。受験表明が存在しない言語/Lvに対して、単独でレビュー依頼を作成することはできない。

### 6.3 評価面談

評価面談応募も受験表明に紐づく子フローとして扱う。面談後の合格判定は、取得資格反映と受験単位のクローズにつながる。

## 7. 機能要件

### 7.1 認証・認可

- ユーザーはメールアドレスとパスワードでログインできる
- パスワードはハッシュ化して保存する
- ログアウトできる
- 管理者、受験者、評価官のロールを持てる
- 画面はCookie/session認証を使う
- APIはJWTアクセストークンを使う
- リフレッシュトークンを発行し、ローテーションできる
- Pundit等でロール、テナント、状態に応じた認可を行う
- 認可失敗はログに残す

### 7.2 受験対象マスタ管理

- 受験対象を一覧表示できる
- 受験対象を登録、編集、無効化できる
- Excel/CSVから受験対象を取り込める
- 取込前にプレビューできる
- 取込エラーを行単位で確認できる
- 受験対象は以下の属性を持つ
  - 技術領域
  - 言語
  - フレームワーク
  - Lv
  - 外部ナレッジ参照URLまたは識別子
  - 表示順
  - 説明
  - バージョン
  - 有効/無効
- 評価基準本文や詳細カテゴリはアプリ内に保持しない
- 外部ナレッジ参照は、レビュー時に評価官が詳細基準を確認するための参照情報として扱う

### 7.3 受験表明・受験対象選択

- 受験者は受験表明を登録できる
- 受験者は受験対象マスタからレビュー対象の言語/Lvを選択できる
- 受験表明はレビュー依頼と面談応募の親単位になる
- 受験表明を登録すると、その受験単位に対するレビュー依頼を作成できる
- 受験表明を登録すると、その受験単位に対する評価面談へ応募できる
- 同一評価期で同じ言語/Lvを重複受験表明できない
- 選択後、提出物登録前は下書き状態になる
- 言語/Lvの選択は受験者本人と管理者のみ編集できる
- 受験者はMarkdown形式の任意コメントを登録できる
- Markdownコメントでは見出し、リスト、リンク、コードブロックを使える
- Markdownはサニタイズして表示し、危険なHTMLやscriptを実行しない
- 受験表明は、表明済み、レビュー中、面談調整中、面談予定、合格、不合格、取消、クローズなどの状態を持つ
- 合格判定後、受験表明はクローズ状態になる

### 7.4 提出物管理

- 受験者は提出物ファイルをアップロードできる
- 受験者はGitHubリポジトリURLを登録できる
- Markdown形式の補足説明・アピールコメントを登録できる
- 提出物種別はファイル、GitHubリポジトリ、補足資料とする
- 添付ファイル名、サイズ、拡張子を検証する
- GitHubリポジトリURLは形式検証する
- Markdownコメントは文字数上限を持つ
- 提出後の変更は履歴として残す

### 7.5 レビュー申請ワークフロー

レビュー申請は以下の状態を持つ。

- 下書き
- 提出
- レビュー中
- 差し戻し
- 修正済み
- 承認
- 却下
- 取消

状態遷移は現在状態、ユーザーロール、評価期によって制御する。

レビュー依頼は受験表明に紐づく。対象の言語/Lvに対応可能な評価スキルを持つ評価官のキューに表示する。対応可能な評価官のうち誰か1人が差し戻し、承認、却下を行った場合、レビュー依頼全体のステータスを更新する。

### 7.6 コメント・レビュー

- 評価官はレビュー依頼に対してコメントできる
- 評価官は差し戻し理由を登録できる
- 評価官は承認コメントを登録できる
- コメント、差し戻し、承認は履歴として残る
- 受験者は自分の申請に対するコメントを確認できる
- 評価官は自分の対応可能評価スキルに該当するレビュー依頼のみ確認できる
- 評価官は他の評価官が既に対応済みのレビュー依頼を重複対応できない

### 7.7 検索・一覧・集計

- 受験対象を技術領域、言語、フレームワーク、Lv、有効/無効で検索できる
- 受験者を検索できる
- レビュー状態で検索できる
- レビューコメント本文で検索できる
- 複数条件検索ができる
- 一覧はページネーションする
- N+1を防ぐ
- 承認率、差し戻し件数、未対応件数を集計できる

### 7.8 通知

- ReviewApplication、InterviewApplication、ExamApplicationの状態変更をStatusChangeEventとして記録できる
- 状態変更イベントを画面上の履歴として確認できる
- レビュー依頼時にSlackへ送信できる
- 差し戻し時にSlackへ送信できる
- 承認時にSlackへ送信できる
- 面談依頼時にSlackへ送信できる
- Slack送信失敗時はSlackDeliveryに失敗履歴を残し、リトライできる
- Slack送信処理は非同期ジョブで実行する

### 7.9 評価面談応募・管理

- 受験者は受験表明に紐づく評価面談応募を登録できる
- 受験表明が存在しない場合、評価面談応募は登録できない
- システムは評価官マスタから面談対応評価官を割り振りフォームへ自動フィルインできる
- 自動フィルインは保存前の初期値であり、最終確定ではない
- 自動フィルインは面談対応数が最も少ない評価官を優先する
- 対象評価官は自動フィルインされた面談対応評価官を変更して確定できる
- 評価官確定後、受験者は希望日時を指定できる
- 評価官は希望日時を承認または差し戻しできる
- 承認後にGoogle Calendar予定を作成する
- 面談結果を保存できる
- 評価官は面談後に合格/不合格を判定できる
- 合格判定時にユーザーの取得資格へ反映できる
- 合格判定時に受験表明をクローズできる

### 7.10 取得資格管理

- ユーザーごとの取得資格を一覧表示できる
- 取得資格は合格判定を起点に作成または更新される
- 取得資格は受験対象、取得日、判定した評価官、根拠となる受験表明を保持する
- 同一ユーザー、同一受験対象の取得資格は重複登録しない
- 受験者は自分の取得資格を確認できる
- 管理者は取得資格の一覧を確認できる

### 7.11 Google Calendar連携

- 面談日時を登録できる
- 受験者と評価官のメールアドレスを使ってGoogle Calendar予定を作成できる
- Calendar登録に失敗した場合はエラーを記録し、再実行できる
- Calendar連携の認証情報は環境変数で管理する

### 7.12 帳票・エクスポート

- 受験対象マスタをExcel出力できる
- レビュー結果一覧をCSV/Excel出力できる
- 受験者別提出状況をCSV/Excel出力できる
- 取得資格一覧をCSV/Excel出力できる
- 大量出力は非同期ジョブ化する

### 7.13 API

- `/api/v1` 配下にREST APIを提供する
- APIはJSONを返す
- APIはJWTで認証する
- APIは統一エラーレスポンス形式を持つ
- APIはバージョニングする
- APIはページネーションを持つ
- OpenAPI定義を用意する
- GraphQL Query/Mutationを用意する
- ActionCableでレビュー通知を配信する
- gRPCは実装せず、`.proto` 例と比較説明をdocsに残す

## 8. 非機能要件

### 8.1 セキュリティ

- 認証が必要な画面/APIは未ログインでアクセスできない
- 受験者は自分の申請以外を編集できない
- 評価官は自分が対応可能な評価スキルに該当するレビュー依頼をレビューできる
- 管理者は全体を管理できる
- テナントスコープを適用する
- SQLインジェクションを避ける
- CSRF対策を有効にする
- session fixation対策を行う
- パスワードは平文保存しない
- 機密値をログに出さない
- ReDoSを避ける正規表現設計と入力長制限を行う

### 8.2 性能

- 一覧画面でN+1を発生させない
- 検索対象カラムには必要なindexを設計する
- EXPLAINで主要クエリを確認できる
- 大量取込は逐次処理またはbatch処理する
- 大量出力は非同期ジョブ化する

### 8.3 可用性・回復性

- 外部API失敗時はリトライする
- リトライしても失敗する場合は失敗状態を記録する
- 非同期ジョブは冪等性を考慮する
- 二重実行で状態が壊れないようにする

### 8.4 保守性

- Controllerに業務ロジックを寄せない
- service/usecase層に処理を分離する
- 外部連携はclientクラスに分離する
- DIで外部依存を差し替えられるようにする
- 値オブジェクトで検索条件、取込行、Calendar payloadを扱う
- Pundit等で認可を集約する

### 8.5 監査性

- 重要操作を監査ログに残す
- 認可失敗をログに残す
- 状態遷移履歴を残す
- コメント履歴を残す
- 提出物変更履歴を残す

### 8.6 論理削除

- 業務データは原則として物理削除しない
- 論理削除は `deleted_at` で表現する
- Rails実装では `paranoia` gemを使う
- 通常の一覧、検索、APIでは論理削除済みデータを表示しない
- 管理者の履歴確認、監査、復旧確認では論理削除済みデータも参照できる
- 業務上の取消、却下、クローズは `status` で表現し、`deleted_at` と混同しない

- REST API、GraphQL、ActionCableの最低限の動作を確認できる
- 主要テストが通る
- 評価基準仮対応表と要件定義書の整合が取れている
